## Source Story

### Story 5.8: Troubleshooting and Failure Reason Docs

As a maintainer or operator,
I want troubleshooting docs tied to validation and workflow failure reasons,
So that update, build, publish, catalog, and security failures can be resolved without guessing.

**Acceptance Criteria:**

**Given** validation and workflow summaries produce structured failure reasons
**When** users read `docs/troubleshooting.md`
**Then** docs cover invalid metadata, stale generated files, unsupported package combinations, wrong tag policy, wrong `latest`, PG19 experimental failures, Docker build failures, PostgreSQL startup failures, SQL smoke failures, missing SBOM/provenance/signature, vulnerability threshold failures, and catalog reference failures.
**And** each troubleshooting entry names the likely command to run locally.
**And** entries distinguish `trixie` primary failures from `bookworm` secondary failures where relevant.
**And** entries explain when a combination should hard-fail versus when it should be marked `publish: false` with `skip_reason`.
**And** troubleshooting docs do not recommend bypassing release-blocking gates for normal releases.

## Troubleshooting Contract

- Troubleshooting docs must map each structured failure reason to a likely local command and remediation.
- Required categories: invalid metadata, stale generated files, unsupported package combinations, wrong tag policy, wrong `latest`, PostgreSQL `19beta1` experimental failures, Docker build failures, PostgreSQL startup failures, SQL smoke failures, missing SBOM/provenance/signature, vulnerability threshold failures, and catalog reference failures.
- Entries must distinguish hard-fail cases from combinations that should be represented as `publish: false` with `skip_reason`.
- Entries must distinguish primary `trixie` and secondary `bookworm` issues where package or catalog behavior differs.
- Docs must not recommend bypassing scan, SBOM/provenance, signing, publish, catalog, permissions, or validation gates for normal releases.
- Failure reason catalog generation must stay deterministic and use UTC dates when dates appear.

## Failure Reason Catalog Contract

- `cloudnative-pg-timescaledb/docs/generated/failure-reason-catalog.md` is generated by `make generate` through `cloudnative-pg-timescaledb/scripts/generate-docs.sh` from workflow/script failure reason definitions and docs templates.
- Each catalog entry must contain: `reason_id`, `category`, `applies_to`, `gate_or_command`, `hard_fail`, `publish_false_skip_reason_allowed`, `local_command`, `remediation`, and `trixie_bookworm_notes`.
- Required `reason_id` values:
  - `metadata.invalid`
  - `generated.stale`
  - `package.unsupported-combination`
  - `tag.policy-invalid`
  - `tag.latest-invalid`
  - `postgresql.pg19-experimental-policy`
  - `build.docker-failed`
  - `runtime.postgresql-startup-failed`
  - `smoke.sql-extension-failed`
  - `evidence.sbom-missing`
  - `evidence.provenance-missing`
  - `evidence.signature-missing`
  - `scan.vulnerability-threshold-failed`
  - `catalog.reference-invalid`
- Generated catalog validation must fail if any required `reason_id` is missing, duplicated, lacks a local command/remediation, or omits the hard-fail versus `publish: false` with `skip_reason` distinction.
- `trixie_bookworm_notes` must explicitly say `same` when the remediation is identical for both Debian variants, or describe the package/catalog difference when it is not.
